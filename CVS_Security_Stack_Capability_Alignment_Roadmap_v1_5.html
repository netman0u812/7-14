Purpose & provenance

This roadmap consolidates every gap identified across four prior-session documents into one prioritized, dependency-sequenced plan: the PA vs. Zscaler DLP & Malware/Threat Equivalency Analysis (v8.1), the Palo Alto Panorama Security Policy Documentation (v8, superseding v5), and the existing PA/Zscaler Functionality Implementation Roadmap (v1.5), which already scoped and sequenced three gaps (Encrypted Egress File Blocking, AI Access Suppression, Browser Blocking). Those three are carried forward here unchanged in substance, folded into the broader sequence rather than re-derived. Nothing in this document re-verifies findings already confirmed in the source material; where a finding is inferred rather than confirmed, it's marked as such.

  • Governance decisions are sequenced before the technical work that depends on them.
  • Foundational/prerequisite work (logging integrity, Cluster B baseline URL filtering) is sequenced before the capabilities that build on it.
  • Effort estimates reflect configuration complexity, not security impact — a low-effort item is not necessarily the highest priority in risk terms, only the fastest to close.

Risk Assessment Methodology — NIST SP 800-30 Rev. 1

Confirmed current: NIST SP 800-30 Revision 1, "Guide for Conducting Risk Assessments" (Sept. 2012) remains the authoritative version — no Revision 2 exists. Applied here as an additional lens layered on top of the phase sequencing above, not a replacement for it: phase order reflects dependency (what has to happen before what), while the risk rating below reflects severity/urgency (how much it matters if a gap stays open). A "Very High" risk item can still sit in a later phase if it's genuinely gated on a governance decision or missing data — the two lenses answer different questions.

SP 800-30's process rates each risk as a function of two factors: likelihood that a threat event exploits a given vulnerability, and impact — the adverse effect on operations, assets, or individuals if it does. Both are rated qualitatively (Very Low / Low / Moderate / High / Very High) and combined into an overall risk level; the guide also accounts for predisposing conditions (organization-specific factors that increase or decrease likelihood, e.g., a healthcare/PBM data footprint raising the stakes of a DLP gap specifically for CVS). Ratings below are qualitative judgments grounded in the confirmed findings in this roadmap — not computed from a formal threat-event catalog, which SP 800-30's full process would require as a separate exercise if a formal risk register is wanted.

Open Data Gaps — Tracked, Not Yet Actionable

PA SSL/TLS Decryption Policy — status unconfirmed. This is foundational, not cosmetic.

Checked exhaustively: the only two mentions of "decryption" anywhere in the Panorama documentation are passing references to it as one of several log types forwarded to Panorama — neither confirms an actual Decryption Policy is configured, nor how much traffic it covers if it is. A PA Decryption Policy is a structurally separate rulebase from Security Policy (Panorama: Policies > Decryption vs. Policies > Security), and none of the four source documents ever examined it — the scope was explicitly "Security Rules (45)."

Why this matters more than a typical open gap: DLP content-pattern matching, deep Threat Prevention signatures, and file-type-based File Blocking generally require visibility into decrypted payload to inspect HTTPS sessions at Layer 7. If PA isn't decrypting broadly, the DLP Alert→Block finding (Phase 2), the Threat Prevention findings (Phase 3), and the File Blocking finding (Phase 1) may all be describing controls that only see a fraction of real-world traffic — the vast majority of which is HTTPS today. This doesn't invalidate those findings (the alert-only/blocking distinction is still real for whatever traffic PA does see), but it means their practical impact can't be fully assessed until decryption coverage is confirmed.

By contrast, Zscaler's SSL inspection is confirmed with concrete evidence — see the corrected SC-8 row above. This asymmetry (one platform confirmed, one not) is itself worth surfacing to whoever owns the PA estate directly.

Requested: a Decryption Policy export from Panorama (rule list, per-rule Decrypt/No-Decrypt/Exclude action, and the SSL Forward Proxy certificate/trust configuration), same rigor as the Security Policy export already analyzed.

McAfee/Skyhigh — no policy data received.

Per direct confirmation, exports aren't available yet. This remains an open gap: McAfee/Skyhigh is understood to cover the same functional ground as Zscaler via the same client-driven traffic steering model, but nothing about its actual configuration — block/alert posture, DLP coverage, threat categories, geo-blocking — has been verified. Until this data exists, any statement about CVS's overall (as opposed to PA/Zscaler-specific) security posture is incomplete — an unknown share of users may be steered through McAfee/Skyhigh instead of Zscaler, and this roadmap cannot speak to their actual protection level. Flagged as a standing prerequisite for Phase 6 below, not guessed at.

Zscaler URL Filtering structural mismatch — RESOLVED as of this revision, not an open gap.

The source PDF (New - Zscaler Cloud Portal - Print All Policies.pdf, 188 pages, the same export cited throughout the Equivalency Analysis) was received in full this session. Direct inspection of pages 24–41 — not previously examined; the prior finding was based on a representative sample that happened not to include this section, not a contradiction of verified fact — confirms Zscaler does have a standalone "URL Filtering Policy" section, distinct from and preceding "Cloud App Control Policy" (which starts page 42). It contains 60 numbered rules. See the corrected Content/URL Filtering findings under Phase 7, which is now resolved rather than blocked. This correction supersedes the "structural mismatch" verdict in the Equivalency Analysis v8.1.

Logging Coverage remediation — inconsistency confirmed, pattern identified

New finding — not previously roadmapped. Refined this revision with rule-by-rule data.
  • Checked log-forwarding assignment across all 45 rules individually, not just the 8 previously flagged. Result: the inconsistency is entirely internal to DG_LV_Egress, not a platform-wide pattern. All 32 rules across the other three device groups (DG_LV_EXT_DMZ, DG_AZ_Egress, DG_RI_Egress) uniformly forward to ForwardtoPanoramaCollector — 100% consistent, no exceptions.
  • Within DG_LV_Egress specifically (13 rules), logging splits along a clear line: the three narrowly-scoped Meritain workgroup rules (Elevated-Rights, HR, IT) correctly forward to ForwardtoPanoramaCollector. Every other rule in the group — the general-purpose Meritain catch-all, the Drop rule, both LV-COLO rules, the YouTube exception, and all three PPOM rules — forwards to the undefined default profile instead. This reads as a rule-by-rule oversight (the specific/sensitive rules were configured correctly, the broader/older ones weren't updated to match) rather than a deliberate design difference — worth confirming with whoever owns DG_LV_Egress rather than assuming.
  • Two rules (LAN-MERITAIN-WG-Allow Adobe, LAN-MERITAIN-TO-INTERNET-MERITAIN-EXCEPTION) have no logging data at all — this is the same 2 rules already flagged as missing from the live config snapshot under config hygiene, not a third category.
  • Confirmed: no rule anywhere in the 45-rule catalog shows any variation in log-start/log-end behavior (uniformly log-start=no, log-end=yes) — the inconsistency is specifically in forwarding destination, not logging behavior itself.
  • Prioritize the Drop rule (LAN-MERITAIN-WG-NO-INTERNET-ACCESS) first — denied-access events for that segment are the ones most likely to matter for incident response, and they're currently in the affected group.
  • Confirm directly on the affected firewalls whether local logging is occurring as a fallback, since this wasn't independently verified beyond the absent profile definition.

  • 14 of 15 rules in the "Best Practice" profile explicitly alert, including every Critical-severity rule for both servers and clients — despite far greater granularity than the alert-only alternative.   • Antivirus, Anti-Spyware, and Vulnerability Protection are alert-only in all 7 profile groups with zero exceptions today — Cluster B has none of the partial blocking Cluster A already has (AV and Spyware Best Practice profiles use reset-both for high-severity threats in Cluster A; Cluster B has nothing equivalent). Phase 7 — Content/URL Filtering: Corrected Verdict (Resolved)

7

Zscaler does have a standalone URL Filtering Policy — the mismatch was a sampling gap, not a real structural difference

Resolved this revision

Confirmed by direct inspection of the full 188-page policy export, pages 24–41: Zscaler's URL Filtering Policy is a distinct, standalone section (separate from Cloud App Control) containing 60 numbered rules. The prior "structural mismatch" finding is superseded, not just supplemented.

Rules 1–59: business-exception model, same pattern already identified elsewhere

  • The large majority are narrowly-scoped named rules (Allow/Block/Isolate/Disabled) tied to specific AD groups, individual users, locations, or time windows — the same exception-driven pattern the Equivalency Analysis found in Cloud App Control. This part of the original finding holds.
  • A notable concentration of these are AI-governance rules — at least 12 identified CVSH-Allow-AI-TempAccess* / CVSH-Temp-AI-Allow rules, each tied to a tracked Enterprise AI Governance request ID and a 90-day access window. Checked against today's date: 11 of 12 have end dates already in the past (the export is dated 5/27/2026; several windows closed as early as May and June 2026). Whether Zscaler auto-disabled these on schedule or they're still live needs direct confirmation — the print export only shows configured state as of the export date, not enforcement history.
  • Rule 43 (TEST-CVSH_AIchatbots_Is...) is an active Block With Redirect rule scoped to DPS_DenyListProd_AIchatbots and the Generative AI category. Rule 44 is its paired Allow-exception, gated on Data Protection Services / Information Governance Board approval. Rule 48 adds an Isolate fallback for the same category, applied to all users. Together these directly answer the previously-open question from the AI Access Suppression item: DPS_DenyListProd_AIchatbots is confirmed enforced, not just a labeled criterion — at the URL-filtering layer specifically, which is a separate control plane from the DLP-layer rules (monitor-only) documented in the Equivalency Analysis. The "TEST-" prefix on rule 43 is worth flagging to the console owner given it's carrying real production blocking.

Rule 60 (G-CVSH-URL Blocked Categories): the missing global block list

  • A genuine, comprehensive category-based Block rule covering roughly 37 categories — Nudity, Pornography, Gambling, Computer Hacking, Copyright Infringement, Militancy/Hate and Extremism, Weapons/Bombs, Spyware/Adware, Remote Access Tools, Custom Encrypted Content, and more — comparable in scope to PA's ~30-category global block profile.
  • Its description states it was added by Guidepoint Security on May 23, 2023, imported from the legacy McAfee proxy configuration — a direct, documented link between McAfee's historical category policy and Zscaler's current one, relevant context for the Phase 6 McAfee/Skyhigh work once that data arrives.

Corrected equivalency verdict

  • Content/URL Filtering is not a structural mismatch — both platforms have a real, comprehensive, global category-block layer (PA's ~30 categories vs. Zscaler's ~37 in Rule 60), plus a large set of narrower business-exception rules layered on top on both sides. A direct side-by-side comparison (category-by-category overlap, similar to the geo-block-list comparison already done) is now possible and should replace the "structural mismatch, not directly comparable" language in the Equivalency Analysis in its next revision.
Dependencies: None remaining — resolved with data already in hand. Recommend a follow-up category-by-category diff against PA's block list as a fast, low-effort addition to Phase 2/3 work rather than a new phase.
