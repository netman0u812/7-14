Key Findings

1. IPS coverage across the confirmed PCI policy set: 48.0%

High Severity
  • Combined across all 18 Tier-1 (explicitly PCI-named) policies: 3,594 of 7,491 enabled Allow rules (48.0%) have an IPS policy attached. 15 of the 18 policies show 0.0% coverage. Only three — PSS-Nexus-PCI-Policy (94.9%), RETAIL-INTPCI-POLICY (55.8%), and the 2-rule GCP-CaaS-PCI Policy (100%) — carry meaningful IPS coverage.
  • The entire Middletown/Windsor Aetna-heritage tier — all 10 policies, 1,562 enabled Allow rules — has confirmed zero IPS coverage, with no mitigating construct found anywhere in the data. This is the single largest, best-supported instance of this gap.
  • The five SHEA/Vegas PSS-LEGACY-PCI-VLAN* policies (2,056 enabled Allow rules) also show 0.0% in their main ACP. A separate "(IPS Only) Las Vegas Voyager" policy exists, with one enabled catch-all rule ("Inspect All", no network scoping, IPS policy "Las Vegas Voyager" attached, mode DETECTION). Whether this applies to the same physical devices enforcing the Legacy VLAN policies could not be confirmed from the data available — FMC's device-to-policy bindings aren't visible in the raw CLI-style device configs, and no reference to this policy name was found in any SHEA device config checked. Requires direct confirmation from the FMC administrator.
  • Hawaii and Pittsburgh have only an "(IPS Only)" construct — no main traffic-control ACP was found for either site across any source provided. This analysis cannot characterize their Allow/Block rule posture, only that a Detection-mode IPS catch-all exists. San Antonio is in the same position.
  • Independent of the binding question: every one of the five "(IPS Only)" policies runs in DETECTION mode, not PREVENTION. Detection mode alerts on intrusions; it doesn't block them. Even in the best-case reading where these are fully bound to the relevant devices, this is a detection layer, not a prevention one.
Sources: Cisco_FMC_PCI/PCI_FW_Policies_All.csv + Cisco_FMC_Aetha/Mid-FMC-PCI-Policies.csv combined for the Tier-1 figure; pazshdc-sec-int-prd-mgt-01_07-08-2026/_IPS_Only__*.csv for the catch-all policies; raw SHEA device configs checked directly for a binding reference and none found.

2. A second, large-scale policy set exists on a PCI-designated FMC manager — included with an explicit confidence caveat

Medium Severity / Scope Question
  • The FMC manager directory rricmdc-sec-mgt-prd-pci-01 — named as a PCI security-management platform — hosts 8 additional policies totaling 25,439 rules: INT-DCOM-Policy (13,886), EXT-DCOM-Policy (5,472), Retail-ATG-INT-Policy (2,819), PSS-ATG-INT_Policy (946), Retail-ATG-EXT-Policy (909), az-htrust-001-acp (622), PSS-ATG-EXT_Policy (537), STRATUS_ACCESS_POLICY (248).
  • Unlike the Tier-1 set, none of these policy names explicitly say "PCI" — rule comments reference Tufin migration tickets and RITM change numbers consistent with the same estate-wide pattern seen elsewhere, but the traffic these specifically govern (DCOM, ATG, Stratus, and an Azure-hosted trust relationship) isn't self-evidently cardholder-data traffic the way RETAIL-INTPCI-POLICY is.
  • Being hosted on a manager explicitly labeled for PCI security management is meaningful signal, but not proof that every rule here is CDE-scoped — treated here as Tier 2: plausible PCI-adjacent scope, not confirmed CDE scope.
  • INT-DCOM-Policy is also notably large and only 40.2% enabled (5,590 of 13,886) — if this does turn out to be CDE-scoped, it would be the single largest rule-hygiene concern in the entire estate, well past the Retail policy's 92.7% disabled ratio in absolute rule count.
  • No IPS-policy field exists in this schema at all, so IPS coverage cannot be assessed for Tier 2 from the data available.
Source: FW_POLICIES/rricmdc-sec-mgt-prd-pci-01/*.csv, all 8 files processed in full.

3. Ruleset hygiene — two large policies dominated by disabled rules

Medium Severity
  • RETAIL-INTPCI-POLICY: 2,686 of 2,898 rules (92.7%) disabled — the finding from v1.0, confirmed unchanged.
  • Across the Middletown/Windsor Aetna tier, disabled-rule ratios run 4.2%–17.5% per policy — real but far smaller than Retail's, worth a lighter-touch review rather than a priority item.
  • If Tier 2's INT-DCOM-Policy is confirmed as CDE-scoped, its 59.8% disabled ratio (8,296 of 13,886 rules) would be the largest hygiene gap in the estate by rule count.
  • The category field remains 100% "--Undefined--" across every Tier-1 rule checked — no rules anywhere in the confirmed PCI scope are organized into named categories.
Source: enabled field grouped by policy, Tier-1 combined dataset and Tier-2 INT-DCOM-Policy.

4. Segmentation, logging, and legacy protocols

Medium Severity
  • 23 enabled Allow rules with any/any network scope (mostly protocol-scoped — ICMP, OSPF), 57 with any destination port, concentrated in the Legacy VLAN policies. Explicit deny rules (e.g. OK_DENIED_INBOUND_CITRIX_HOSTS) confirm deliberate design intent exists alongside these.
  • Device-level syslog forwarding is genuinely active — multiple redundant hosts confirmed directly in raw configs at SHEA, Middletown, and Windsor. Connection-start vs. connection-end capture is inconsistent between policy sets: the master set logs 82.1% start / 21.5% end, while the Aetna tier logs 24.3% start / 92.4% end.
  • SNMPv2c community-string authentication is confirmed alongside SNMPv3 on PCI devices across virtually every site checked.
  • TLS 1.2 confirmed where the setting was visible (Woonsocket, GCP) — not confirmed estate-wide.
Source: combined Tier-1 dataset (10,531 rules) and direct device-config review.
